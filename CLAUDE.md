# Главные правила

## Правило №1

Не ломать существующий функционал.

Перед изменениями проверить:

- какие страницы используют код;
- какие компоненты зависят от него;
- какие API используют данный функционал.

---

## Правило №2

Минимальные изменения.

Запрещено:

- переписывать большие блоки без необходимости;
- менять архитектуру ради красоты кода;
- заменять рабочие решения на новые без причины.

---

## Правило №3

Сначала анализ.

Перед любыми изменениями предоставить:

### Что будет сделано

### Какие файлы будут изменены

### Какие риски существуют

### Как откатить изменения

Только потом выполнять задачу.

# Работа с кодом

## TypeScript

Всегда:

- строгая типизация;
- избегать any;
- использовать интерфейсы и типы.

Плохо:

```ts
const user: any
```

Хорошо:

```ts
interface User {
  id: string
  name: string
}
```

---

## React

Использовать:

- функциональные компоненты;
- hooks;
- переиспользуемые компоненты.

Избегать:

- дублирования логики;
- копирования JSX.

---

## Next.js

Предпочитать:

- Server Components;
- Server Actions;
- кеширование там, где возможно.

Не добавлять клиентские компоненты без необходимости.

# Работа с календарём

Календарь является критически важным модулем.

При изменениях:

- не ломать существующие записи;
- сохранять совместимость;
- проверять отображение на мобильных устройствах.

---

# Работа с бронированием

Критические сценарии:

1. Создание записи.
2. Перенос записи.
3. Отмена записи.
4. Уведомления.
5. Работа временных зон.

Любые изменения тестировать отдельно.

- не хранить данные карт;

# Git Workflow

## Гілки

- `main` — стабільний реліз, деплоїться на продакшн для тестів. Не чіпати без причини.
- `dev` — всі нові правки та фічі йдуть сюди.

**Завжди комітити в `dev`:**

```bash
git checkout dev
git add .
git commit -m "опис змін"
```

**Оновити реліз (злити dev → main і задеплоїти):**

```bash
git checkout main
git merge dev
firebase deploy --only hosting
git checkout dev
```

**Перед великою зміною — окрема гілка від dev:**

```bash
git checkout -b feature/new-feature
```

# Версионирование деплоя

После каждого пуша в master обязательно:

1. Написать в чате номер обновления в формате: `Оновлення #N`
2. Обновить константу `APP_VERSION` в `src/App.jsx` — она отображается внизу справа в навбаре
3. Формат версии: `vДД.ММ.N` где N — порядковый номер обновления за день

Пример: `v18.06.1`, `v18.06.2`, `v19.06.1`

Цель: пользователь видит версию в UI и понимает, прошёл деплой или нет.
